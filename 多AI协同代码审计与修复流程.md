### 多AI协同代码审计与修复全流程指南（v3.1）

本流程通过七阶段闭环实现任意项目的代码审计与修复。所有AI角色按职责命名，严禁混用。**所有AI在每次输出时必须引用当前代码的git commit hash，未锚定版本的输出视为无效。**

#### 阶段零：审计维度规划
-   **执行角色**：**Planner（规划师）**
-   **输入**：《详细设计文档》《通信协议》、项目技术栈说明、业务领域描述
-   **核心目标**：动态生成适配当前项目的审计维度
-   **硬性约束**：
    -   每个维度对应的关键检查项不超过8-12个，确保单次会话可完成审查而不截断
    -   每个维度必须输出**具体检查规则**，而非仅输出维度名称
    -   若设计文档信息不足，Planner必须标注`[待验证]`，不得用常识补全
-   **输出**：《审计维度清单》（含维度名、检查规则、所需文档标签）
-   🔴 **人工确认点**：人工审核维度清单的完整性与粒度，确认后进入下一阶段

#### 阶段一：文档智能拆分
-   **执行角色**：**DocSplitter（文档拆解师）**
-   **输入**：原始文档全文 + 《审计维度清单》
-   **核心目标**：将文档拆分为带标签的语义完整片段
-   **硬性约束**：
    -   采用**滑动窗口式拆分**，以"模块-接口"或"功能点"为最小单元，保持语义完整性
    -   每个片段附带**摘要、关键词、溯源指针**（原文章节号）
    -   同一段落可重复打标归属多个维度，但不得切割句子级碎片
-   **输出**：带标签的文档片段库

#### 阶段二：全局侦察与风险地图生成
-   **执行角色**：**Scout（侦察兵）**
-   **输入**：项目文件树 + 关键入口/配置/顶层模块文件 + 文档片段库
-   **核心目标**：生成可验证的风险地图
-   **硬性约束**：
    -   **分两步执行**：
        1.  先扫描项目结构，强制输出《代码结构概览》作为中间产物，包含：模块/包划分、对外暴露的API/路由、核心数据流路径、第三方依赖列表、复杂度预警点
        2.  再结合文档片段库与结构概览进行风险研判，生成风险地图
    -   风险地图每条结论必须附带**代码位置证据**（类名、函数名、行号范围）
    -   风险地图总字数≤2000字；《代码结构概览》不计入此限制
-   **输出**：《代码结构概览》+《风险地图》
-   🔴 **人工确认点**：人工快速核验风险地图的证据真实性，确认后进入下一阶段

#### 阶段三：上下文组装与并行审计
-   **执行角色**：**ContextAssembler（上下文装配师）** + 多个 **Audit-{维度名}（审计员）**
-   **ContextAssembler硬性约束**：
    -   组装"共享底座（风险地图+代码结构概览）+ 专属层（对应标签文档片段）"
    -   执行**一致性校验**：若文档片段与风险地图冲突，明确标注并以设计文档为基准，风险地图仅作"疑似"提示
-   **各Audit-{维度名}硬性约束**：
    -   输出结构必须统一：**问题代码位置 + 违反的具体规则（引用Planner检查清单条目） + 修复建议 + 信心等级（确定/推测/待验证）**
    -   信心等级为"推测"或"待验证"的问题单独标记，不影响批次划分
-   **输出**：结构化问题列表 → 汇总后按严重程度划分为三个批次

#### 阶段四：分批修复与定向验证闭环
按批次（严重→中等→低优先级）严格执行以下四步：

1.  **修复**：**Fixer（修复师）** 接收问题+相关文档片段，完成修改。**强制输出"影响分析"**（修改了哪些接口、需同步更新的调用方列表）
2.  **自动化兜底**：编译+单元测试全绿方可继续。未通过则打回Fixer
3.  **定向复核**：由**原Audit-{维度名}** 复核。输入为：修复后代码 + 原始问题描述 + Fixer的影响分析。复核范围**限定于影响分析所列的相邻代码**，不得无限扩展
4.  **冲突仲裁**：若不同Audit AI对同一修改产生分歧，由**Arbiter（仲裁官）** 裁决。仲裁优先级显式化为：**功能正确性 > 安全性 > 性能 > 可读性**。Arbiter必须引用具体规则并给出推理链

当前批次全部闭环后，方可开启下一批次。

#### 阶段五：全量收尾审计
-   **执行角色**：所有 **Audit-{维度名}**
-   **输入**：最终版代码 + 风险地图 + 调整后的提示词
-   **核心目标**：双目标扫荡——**新引入问题优先** + **前几轮可能遗漏的深层盲区补充**
-   **提示词要求**："代码已完成多轮修复，请基于完整上下文对所有维度进行最终审查。优先检查修复链中的耦合与边界条件导致的新问题，同时关注此前审计中因上下文限制可能未覆盖的深层逻辑缺陷。"
-   **输出**：新增/遗漏问题列表 → 若有，回到阶段四；若无，流程结束

#### 贯穿全程：元认知监控
-   **执行角色**：**Monitor（监控官）**
-   **职责**：不参与具体审计，定期检查各AI输出是否符合格式规范、是否出现幻觉循环或格式崩溃。有权触发重试或降级处理
-   **触发条件**：连续两次输出格式不合规、同一问题三轮未闭环、信心等级分布异常

### AI角色速查表（v3.1）

| 角色命名 | 中文称谓 | 核心职责 | 出现阶段 |
| :--- | :--- | :--- | :--- |
| Planner | 规划师 | 动态生成审计维度+检查规则 | 阶段零 |
| DocSplitter | 文档拆解师 | 滑动窗口拆分+溯源指针 | 阶段一 |
| Scout | 侦察兵 | 结构概览+文档研判→风险地图 | 阶段二 |
| ContextAssembler | 上下文装配师 | 分层组装+一致性校验 | 阶段三 |
| Audit-{维度名} | 审计员 | 单维度审查+结构化输出 | 阶段三/四/五 |
| Fixer | 修复师 | 代码修复+影响分析 | 阶段四 |
| Arbiter | 仲裁官 | 显式规则裁决+推理链 | 阶段四 |
| Monitor | 监控官 | 格式守护+异常熔断 | 全程 |
