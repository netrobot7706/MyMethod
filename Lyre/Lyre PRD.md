# **Lyre项目产品需求文档**
---

## 1. 项目概述

### 1.1. 项目背景与目标
Lyre 是一款为音乐创作者设计的便携式 MIDI 控制器。它配备4个100mm行程推杆，旨在提供精确、顺滑的实时控制体验。项目目标是打造一个软硬件高度协同、配置灵活、性能可靠的创作工具，从根本上杜绝信号抖动和误触发等常见控制器痛点。

### 1.2. 产品定义
Lyre 是一个带有四个 10CM 行程推杆的 USB MIDI 控制器，允许用户通过物理推杆发送 MIDI CC 控制信号。

### 1.3. 核心功能
- **MIDI 控制**: 4个推杆发送可配置的 MIDI CC 消息。
- **低噪声高精度**: 硬件电源滤波与 Control Surface 库原生滤波能力双重保障，确保信号纯净，物理行程完美映射到 MIDI 0-127 全范围。
- **灵活配置**: 用户可通过 Web 上位机独立配置每个推杆的 CC 号和 MIDI Channel。
- **配置持久化**: 设备可本地存储用户配置，支持临时测试和永久保存模式。设备与上位机之间保持配置状态同步。
- **设备校准**: 上位机支持自动捕获推杆行程最值，完成高精度校准。
- **硬件交互反馈**: 利用板载 WS2812 LED 提供设备状态和操作的视觉反馈。

---

## 2. 硬件规格

### 2.1. 核心组件
- **主控芯片**: RP2040-Zero 开发板。
- **输入设备**: 4个 100mm 行程线性电位器（推杆）。
- **视觉反馈**: 板载 WS2812 RGB LED（固件可编程控制）。

### 2.2. 电路设计要点
- **电源方案**: RP2040-Zero 的 `3V3` 输出经 `22R` 电阻限流及 `10µF`、`0.1µF` 电容滤波后，生成纯净电源 `V_clear`，**专用于** 4个电位器供电。
- **输入电路**: 电位器抽头信号需经 `1kΩ` 电阻和 `0.1µF` 电容组成的低通 RC 滤波器后，接入 RP2040 的 ADC 引脚。
- **引脚分配**:
  - 推杆 1 ADC 输入: `GPIO26` (A0)
  - 推杆 2 ADC 输入: `GPIO27` (A1)
  - 推杆 3 ADC 输入: `GPIO28` (A2)
  - 推杆 4 ADC 输入: `GPIO29` (A3)
  - WS2812 LED 数据引脚: 使用 RP2040-Zero 板载默认引脚（一般为 `GPIO16`，以实际开发板为准）。
- **结构要求**: 外壳需在 WS2812 LED 位置开孔或设计导光柱，确保灯光可见。

## 3. 固件功能需求 (Firmware)

### 3.1. 技术栈与环境
- **IDE**: Arduino IDE
- **核心库**:
  - `TinyUSB`: 实现设备枚举。
  - `LittleFS` (或等效 EEPROM 模拟库): 用于配置文件的本地持久化存储。
  - `Adafruit NeoPixel` 或等效库: 用于板载 WS2812 LED 控制。

### 3.2. USB 设备枚举
- 设备枚举USB MIDI设备。
- USB VID/PID 待定，可先使用测试用 VID/PID，最终需申请或购买专用 VID/PID。
- **MIDI 类兼容性约束**：
  - USB MIDI 描述符必须包含符合 USB MIDI v1.0 规范的 **MIDI Streaming 接口描述符**，其中至少包含一个嵌入式虚拟 MIDI IN Jack（对应设备 → 主机的 MIDI 数据流）和一个外部虚拟 MIDI OUT Jack（对应主机 → 设备的 MIDI 数据流）。
  - **说明**：TinyUSB 的默认 MIDI 配置已包含标准 Jack 描述符，正常使用即可满足。本约束旨在确保在后续自定义 USB 描述符、更换设备名称或 VID/PID 时，不会无意中丢失或破坏 Jack 拓扑结构，导致部分 DAW（如 Logic Pro 等）无法识别设备。
  - **验收标准**：设备在 macOS 的「音频 MIDI 设置」中应正确显示为 MIDI 设备，端口路由正常；在主流 DAW（Ableton Live、Logic Pro、Cubase、FL Studio 等）的 MIDI 端口列表中可见且可正常收发数据。

### 3.3. 数据采集与处理（信号链）

| 验收项 | 标准 | 测试方法 |
| :--- | :--- | :--- |
| **静止稳定性** | 推杆在任意位置静止不动时，**零** MIDI 消息输出。 | 将推杆置于中部，保持5分钟，记录MIDI消息数。 |
| **端点覆盖** | 推杆推至物理最底端时，稳定输出 MIDI 值 **0**；推至物理最顶端时，稳定输出 MIDI 值 **127**。 | 手动推至两端，重复10次，每次都能达到0或127。 |
| **响应灵敏度** | 推杆移动时，MIDI 值变化顺滑且及时，无明显的"台阶感"或"跳跃感"。 | 缓慢匀速推动推杆，观察上位机数值变化是否连续。 |

### 3.4. MIDI 消息发送逻辑
- **状态变化触发（由库内置保证）**
- **消息格式**: `Bn CC vv` (n = channel-1, CC = 控制器号, vv = 当前值)

### 3.5. 上位机通信协议
【由Jigsaw协议（MIDI cc上的透明字节流）、Veloce厂牌基本命令协议、Lyre产品特有命令协议构成。参看对应文档】

### 3.6. 本地配置与存储

#### 3.6.1. 默认出厂配置
- 推杆 1: CC#1, Channel 1
- 推杆 2: CC#2, Channel 1
- 推杆 3: CC#3, Channel 1
- 推杆 4: CC#4, Channel 1
- 校准值: Min=0, Max=4095 (12-bit ADC 理论值，可根据实际调整)。

#### 3.6.2. Flash写入策略
写 Flash 操作**采用同步阻塞方式**，即收到指令后直接执行写入，写入完成后返回应答。

```
设计决策说明：
用户在执行配置保存操作时，注意力在上位机界面，不会同时演奏设备。
Flash 写入的阻塞时间通常在 200ms 以内，对用户体验无实质影响。
保持同步写入方案可以简化固件与上位机的架构，避免不必要的异步状态机复杂性。
```

实现要求：
- 收到写 Flash 指令后，在当前位置直接执行写入操作。
- 写入期间（通常 < 200ms）暂停 ADC 读取和 MIDI 消息发送，这是可接受的。
- 写入完成后立即返回应答。
- 代码中需保留注释，标明此为**有意为之的设计决策**，而非疏忽。


### 3.7. LED 交互反馈规范
使用 RP2040-Zero 板载的 WS2812 LED 提供设备状态视觉反馈。

**LED 状态定义表**：

| 设备状态/事件 | LED 行为 | 颜色 | 说明 |
| :--- | :--- | :--- | :--- |
| **USB枚举成功/空闲** | 呼吸（缓慢明暗交替） | 白色 (#FFFFFF) | 设备正常工作，等待操作。 |
| **MIDI 消息发送** | 快闪一次（50ms） | 绿色 (#00FF00) | 每次发送有效 MIDI CC 消息时触发。 |
| **进入测试模式** | 持续闪烁（500ms周期） | 紫色 (#8000FF) | 收到 `test_config` 后，提示用户当前为未保存的临时配置。 |
| **保存配置中** | 快速闪烁（200ms周期） | 黄色 (#FFAA00) | 执行 `save_config` 写入 Flash 期间。 |
| **保存配置完成** | 长亮2秒 | 绿色 (#00FF00) | `save_config` 完成，随后恢复空闲状态。 |
| **恢复出厂中** | 快速闪烁（100ms周期） | 红色 (#FF0000) | 执行 `factory_reset` 写入 Flash 期间。 |
| **恢复出厂完成** | 长亮2秒 | 绿色 (#00FF00) | `factory_reset` 完成，随后恢复空闲状态。 |
| **固件升级模式** | 交替闪烁 | 红色/蓝色交替 | （预留）进入 Bootloader 或固件升级状态。 |

**LED 优先级**：
当多个 LED 触发条件同时满足时，按以下优先级显示：
1. 固件升级模式（最高）
2. 保存配置中/完成
3. 恢复出厂中/完成
4. 测试模式
5. MIDI 消息发送（与空闲状态共享时间片，MIDI 快闪覆盖呼吸状态）
6. USB枚举成功/空闲（默认状态，最低）

## 4. 上位机功能需求 (Web App)

### 4.1. 技术栈
- `Vue 3` + `TypeScript` + `Tailwind CSS` + `Web Serial API`

### 4.2. 通信与连接管理
- **UI 组件**: 设备选择下拉框、连接/断开按钮、连接状态指示灯。

- **串口选择策略**:
  - 使用 Web Serial API 的 `requestPort()` 方法弹出浏览器原生串口选择窗口。
  - 系统应通过 USB 的 `vendorId` 和 `productId` 进行预过滤（待固件 VID/PID 确定后实现），尽量在列表中就减少用户误选的可能性。

- **连接与握手流程**:
  1. 用户点击"选择设备"按钮，浏览器弹出串口选择窗口。
  2. 用户选择串口并授权后，上位机尝试打开该串口。
  3. 串口打开成功后，上位机**立即发送** `{"cmd":"handshake"}` 指令。
  4. 上位机启动一个 2 秒超时计时器，等待设备应答。
  5. **成功路径**：在超时前收到有效 `handshake_ack` 应答：
     - 状态指示灯变**绿**。
     - 连接按钮文字变为"断开连接"。
     - 上位机**依次发送** `read_config` 和 `get_runtime_config`，获取设备的持久化配置和当前运行时配置。
     - 比较两者：若一致，UI 正常显示；若不一致，UI 做出视觉提示（见4.3节）。
  6. **握手失败/超时路径**：
     - 状态指示灯变**红**。
     - 弹出明确的错误提示框：
       > **"设备连接失败"**
       > 
       > 未能在此串口上检测到 Lyre 设备。可能的原因：
       > 1. 选择的串口不正确（请尝试其他串口）。
       > 2. 设备未正确连接或固件未正常运行。
     - 上位机自动关闭当前串口，UI 恢复为未连接状态，允许用户重新选择。

- **运行时状态同步**：
  - 每次执行 `test_config` 并收到 `ack` 后，上位机自动调用 `get_runtime_config` 刷新 UI。
  - 设备断开后重新连接时，同样执行上述比较逻辑。

- **其他异常处理**:
  - 浏览器不支持 Web Serial API 时，在连接区域显示"您的浏览器不支持 Web Serial API，请使用 Chrome 或 Edge 等兼容浏览器"的友好提示。
  - 通信过程中串口意外断开（如拔出USB），状态指示灯变红，弹出提示"设备已断开连接"，UI 恢复为未连接状态。

### 4.3. 配置界面
- **布局**: 4个并排或田字格排列的"推杆配置卡片"。
- **每个卡片包含**:
  - **推杆标识**: "推杆 1" 至 "推杆 4"。
  - **CC# 设置**:
    - **快捷语义选择**：提供一个下拉菜单，列出标准 MIDI CC 控制器语义（完整清单见附录A）。
      - 菜单分为两组：「常用」组排在前面，「全部」组按 CC# 编号顺序列出。
      - 当用户选择一个语义时，CC# 输入框自动填入对应编号。
    - **手动输入**：语义菜单旁边保留一个数字输入框（范围 `0-127`），允许用户手动输入任意 CC#。当用户手动修改数字时，语义菜单自动切换至"自定义"状态。双向同步，互不冲突。
  - **Channel# 设置**: 数字输入框或下拉选择，范围 `1-16`。
  - **"临时测试"按钮**:
    - 点击后，仅将该卡片的配置以 `test_config` 指令发送给设备。
    - 发送成功后，上位机自动调用 `get_runtime_config` 刷新 UI。
    - 此操作不写入固件 Flash，断电即失效。
  - **"未保存"状态指示**：
    - 当某个推杆的运行时配置与持久化配置不一致时，该推杆卡片边框变为**橙色虚线**，顶部显示**"未保存"**标签。
    - 当所有配置一致时，视觉提示消失。

### 4.4. 全局操作
- **"保存到设备"按钮**: 放置在页面底部醒目位置。
  - 点击后，收集4个卡片的所有配置，打包为 `save_config` 指令发送。
  - 按钮变为**"保存中..."**并禁用。
  - 收到 `{"status":"ok"}` 后，弹出"配置已永久保存！"成功提示，按钮恢复。
  - 若 3 秒超时，弹出错误提示并恢复按钮。
  - 成功后清除所有卡片的"未保存"标识。

- **"恢复出厂设置"按钮**:
  - 点击后弹出二次确认框，确认后发送 `factory_reset` 指令。
  - 按钮变为**"恢复中..."**并禁用，3 秒超时处理。
  - 成功后调用 `read_config` 和 `get_runtime_config` 刷新 UI 并清除"未保存"标识。

### 4.5. 校准功能（动态自动捕获模式）

#### 4.5.1. 入口
- 在每个推杆配置卡片上，提供"校准"按钮或图标。

#### 4.5.2. 校准交互流程
1. 用户点击某推杆的"校准"按钮，弹出校准向导对话框。
2. 向导显示提示文案：
   > **"请将推杆从最底端匀速推至最顶端，反复 2-3 次。"**
3. 用户点击 **"开始捕获"** 按钮。
4. 上位机进入**连续读取模式**：以一定频率（建议每 50ms 一次）循环发送 `{"cmd":"get_adc","ch_num":X}` 指令。
5. 上位机实时显示 ADC 值，并动态更新检测到的 Min 和 Max。
6. 当 Min 和 Max 在连续多次采样中趋于稳定（如最近10次采样的波动 < 5 个ADC单位），自动提示：
   > **"校准完成！检测到范围：Min = 205, Max = 3890"**
7. 用户可选择：
   - **"保存校准值"**：发送 `{"cmd":"calibrate",...}` 将 Min/Max 写入 Flash。
   - **"重新捕获"**：丢弃当前结果，重新开始。
   - **"取消"**：放弃校准，关闭对话框。

#### 4.5.3. 边界与异常处理
- 若 Min 和 Max 差值过小（如 < 500 ADC单位），提示"检测不到有效行程，请确保推杆活动范围正常后重试"。
- 若连续读取超时（如 3 秒无新数据），提示"通信中断，请检查设备连接后重试"。
- 校准完成后，卡片上显示"已校准 ✓"标识及校准后的有效范围。

## 5. 非功能性需求
- **响应速度**: 推杆移动至 MIDI 信号输出的端到端延迟应 < 10ms。
- **鲁棒性**: 在未连接上位机时，设备应能独立正常工作，使用上次保存或默认配置。
- **易用性**: 上位机 UI 需简洁直观，操作流程符合音乐人直觉。CC 语义化选择能显著降低新手用户的学习成本。
- **状态一致性**: 上位机 UI 与设备运行状态必须保持同步，尤其是测试未保存场景下的视觉提示。
- **兼容性**: 设备应在主流 DAW（Ableton Live、Logic Pro、Cubase、FL Studio 等）中正确识别并正常工作。
