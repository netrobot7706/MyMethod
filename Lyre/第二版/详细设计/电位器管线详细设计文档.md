## 电位器管线详细设计文档

> **文档版本**：v1.0  
> **对应产品**：Lyre MK2 4-推杆 MIDI 控制器  
> **所属管线**：Potentiometer  
> **依赖架构**：《Lyre MK2 产品架构设计文档 v2.2》  
> **遵循规范**：《信息管线星型架构 v1.1》

---

### 1. 模块概述

电位器管线负责 4 路 100mm 推杆的完整信号处理链：从 ADC 硬件采样到 MIDI 事件生成。管线内部严格遵循 HAL → CORE → APP 三层结构，满足架构文档的隔离要求。

- **HAL 层**：封装 RP2040 ADC 多通道轮询，实现带丢弃样本的滑动平均采样。
- **CORE 层**：实现纯算法性的迟滞比较与死区处理，零外部依赖。
- **APP 层**：组合校准映射、变化检测，通过市场 API 对外提供事件查询和控制接口。

### 3. 硬件抽象层设计（HAL）

#### 3.1 职责
- 管理 4 路 ADC 引脚（GPIO26–29）及 RP2040 ADC 外设。
- 实现算法要求的“切换通道后连续采样 N+2 次，后 N 次算术平均”的采样策略。
- 为上层提供单通道原始采样值接口。

#### 3.2 宏配置
```c
#define POT_HAL_SAMPLE_COUNT  16   // N，默认 16
#define POT_HAL_DISCARD       2    // 丢弃的前 2 次采样
```

#### 3.3 内部接口
```c
// 初始化 ADC 硬件及 GPIO
void pot_hal_init(void);

// 对指定通道执行一次滤波采样，返回 12-bit ADC 原始值（0–4095）
// 该函数内部完成通道切换、连续 N+2 次采样、丢弃前 2 次、后 N 次平均
uint16_t pot_hal_read_channel(uint8_t channel);
```

#### 3.4 采样算法细节
1. 选择 ADC 输入通道（`adc_select_input()`）。
2. 连续启动 18 次 ADC 转换（N+2 = 18）。
3. 丢弃前 2 次结果（使采样保持电路稳定）。
4. 将后 16 次结果累加并计算算术平均，四舍五入。
5. 返回平均后的 12-bit 值。

**执行时间估算**：RP2040 ADC 单次采样约 2 μs（96 MHz 主频下），18 次采样合计约 36 μs，满足 10ms 周期内完成 4 路采样的要求。

#### 3.5 实现伪代码
```c
uint16_t pot_hal_read_channel(uint8_t channel) {
    adc_select_input(channel + 26 - 26); // 映射到 ADC 通道号
    uint32_t sum = 0;
    for (int i = 0; i < POT_HAL_SAMPLE_COUNT + POT_HAL_DISCARD; i++) {
        uint16_t val = adc_read();
        if (i >= POT_HAL_DISCARD) {
            sum += val;
        }
    }
    return (uint16_t)((sum + POT_HAL_SAMPLE_COUNT / 2) / POT_HAL_SAMPLE_COUNT);
}
```

### 4. 核心算法层设计（CORE）

#### 4.1 职责
CORE 层实现与硬件无关的迟滞比较与变化判定逻辑，可在不同产品间直接拷贝复用。

#### 4.2 状态数据结构
每个推杆维护一个独立的状态记录：
```c
typedef struct {
    uint16_t last_raw;       // 上次实际发送 MIDI 时的原始值
    uint8_t  last_midi;      // 上次发送的 MIDI 值 (0–127)
} pot_core_state_t;
```

#### 4.3 宏配置
```c
#define POT_CORE_HYSTERESIS  16   // 迟滞区 M（原始值单位），可配置
```

#### 4.4 内部接口
```c
// 初始化一个通道的 CORE 状态
void pot_core_init(pot_core_state_t *state);

// 对原始值进行迟滞比较，若变化足够大则返回 true 并通过输出参数返回新 MIDI 值
// raw_in:       当前滤波后的原始值
// cal_min, cal_max:  校准边界（14-bit 原始值）
// state:        该通道的状态记录（会更新）
// midi_out:     输出参数，仅当返回 true 时有效
// 返回:         是否需要发送新 MIDI 事件
bool pot_core_process(uint16_t raw_in,
                      uint16_t cal_min, uint16_t cal_max,
                      pot_core_state_t *state,
                      uint8_t *midi_out);
```

#### 4.5 迟滞与映射算法
1. **迟滞判断**：  
   若 `|raw_in - state->last_raw| > (POT_CORE_HYSTERESIS / 2)`，则认为推杆发生实质性移动，启动 MIDI 值重算；否则直接返回 false，不更新状态。
2. **线性映射**（原始值 → MIDI）：  
   将原始值限制在 `[cal_min, cal_max]` 内，使用线性插值映射到 `[0, 127]`，结果四舍五入。  
   公式：  
   ```c
   if (raw_in <= cal_min) midi = 0;
   else if (raw_in >= cal_max) midi = 127;
   else midi = (uint8_t)(((uint32_t)(raw_in - cal_min) * 127 + (cal_max - cal_min) / 2) / (cal_max - cal_min));
   ```
3. **MIDI 变化检测**：  
   若计算出的 MIDI 值与 `state->last_midi` 不同，则更新状态（保存新的 last_raw 和 last_midi），并通过 `midi_out` 输出新值并返回 true；否则返回 false，状态不更新。

**设计要点**：  
- 迟滞使用原始值，而非 MIDI 值，避免在映射边界附近因量化抖动导致重复发送。
- 仅当 MIDI 值真正发生变化时才更新 last_raw，确保迟滞以“上次有效发送位置”为基准，防止微小摆动被累积放大。

#### 4.6 重置接口
```c
// 强制清除历史值（校准或配置变更后调用）
void pot_core_reset(pot_core_state_t *state);
```
复位后 `last_raw` 设为 `0xFFFF`，`last_midi` 设为 `0xFF`，确保下一次任意原始值都能触发 MIDI 事件。

### 5. 应用层设计（APP）

#### 5.1 职责
- 组合 HAL 和 CORE 完成 4 通道的全流程处理。
- 通过市场 API `cmd_cfg_api.h` 获取各通道的校准值和 MIDI 映射。
- 管理暂停/恢复和事件缓存。
- 实现 `pot_api.h` 中所有接口。

#### 5.2 内部数据结构
```c
static struct {
    bool paused;                     // 暂停标志
    bool event_pending[POT_COUNT];   // 各通道是否有待读取的 MIDI 事件
    uint8_t event_channel[POT_COUNT];
    uint8_t event_cc[POT_COUNT];
    uint8_t event_value[POT_COUNT];
    pot_core_state_t core_states[POT_COUNT];
} pot_ctx;
```

#### 5.3 对外 API 实现（关联 `market/pot_api.h`）

##### `pot_init()`
- 调用 `pot_hal_init()` 初始化 ADC。
- 遍历 4 个通道，调用 `pot_core_init(&pot_ctx.core_states[i])`。
- 将 `paused` 置 false，清除所有事件标志。

##### `pot_poll()`
由主循环每 10ms 调用一次，内部流程：
```c
void pot_poll(void) {
    if (pot_ctx.paused) return;

    uint16_t raw[POT_COUNT];
    // 1. HAL 采样
    for (int i = 0; i < POT_COUNT; i++) {
        raw[i] = pot_hal_read_channel(i);
    }

    // 2. 查询配置快照（通过市场 API，保证数据一致性）
    pot_mapping_t mappings[POT_COUNT];
    pot_calibration_t cals[POT_COUNT];
    uint8_t count;
    config_get_all_pot_mappings(mappings, &count);
    config_get_all_calibrations(cals, &count);

    // 3. CORE 处理 + 事件缓存
    for (int i = 0; i < POT_COUNT; i++) {
        uint8_t midi_val;
        if (pot_core_process(raw[i], cals[i].cal_min, cals[i].cal_max,
                             &pot_ctx.core_states[i], &midi_val)) {
            pot_ctx.event_pending[i] = true;
            pot_ctx.event_channel[i] = mappings[i].channel;
            pot_ctx.event_cc[i] = mappings[i].cc;
            pot_ctx.event_value[i] = midi_val;
        }
    }
}
```
- 注意：`pot_poll` 每次均重新获取配置快照，可确保配置热更新后立即生效。
- 暂存事件采用最终值覆盖语义，符合 CC 绝对值特性。

##### `pot_get_midi_event(uint8_t index, uint8_t *channel, uint8_t *cc, uint8_t *value)`
- 检查 `pot_ctx.event_pending[index]`。
- 若为 true，则将缓存的值拷贝到输出参数，清除标志，返回 true。
- 否则返回 false。

##### `pot_get_raw(uint8_t index)` 与 `pot_get_all_raw(uint16_t *raw_values, uint8_t count)`
- 如果管线处于暂停状态（`pot_ctx.paused`），返回 0xFFFF。
- 否则立即调用 `pot_hal_read_channel(index)` 并返回，期间需通过全局互斥（例如主循环不并发调用的约定）确保不与 `pot_poll` 冲突。根据单核环境，由调用时序保证互斥，`pot_get_raw` 仅在 `cmd_cfg_task` 的 IDLE 状态调用，此时不会并发执行 `pot_poll`。

##### `pot_set_pause(bool pause)`
- 设置 `pot_ctx.paused`。
- 恢复时不清除历史状态，但若在暂停期间推杆被移动，下次 `pot_poll` 会立即检测出差异并生成事件。

##### `pot_reset_stable_values(void)`
- 遍历 4 个通道，调用 `pot_core_reset(&pot_ctx.core_states[i])`。
- 清除所有事件标志，避免残留事件在恢复后误发。

---

### 6. 并发与数据一致性

- 全管线运行在单核主循环中，无中断抢占，所有全局状态均为自然原子访问。
- `pot_poll()` 是唯一会修改 CORE 状态和事件缓存的函数，主循环保证在轮询-读取事件窗口内不进行其他修改。
- `pot_get_raw()` 与 `pot_poll()` 的互斥由主循环调度保证（0x11 命令仅在 `CFG_IDLE` 时处理，此时 `pot_poll` 与 `pot_get_raw` 不会并发）。

---

### 7. 性能指标

- 单路 HAL 采样时间：~36 μs（18 次 ADC 转换）。
- 4 路全流程（采样+映射+迟滞）：< 200 μs。
- 端到端延迟（物理推杆移动 → MIDI CC 入队）：< 10ms，主要由 10ms 轮询周期决定，处理耗时可忽略。

---

### 8. 可测试性

- CORE 层完全无 HAL 依赖，可独立进行单元测试，验证迟滞边界、线性映射精度、反复震荡抑制。
- HAL 层可通过模拟 ADC 读取函数进行测试。
- APP 层测试时可注入模拟的 CORE 和配置 API，验证事件缓存、暂停/恢复逻辑。
