# Lyre 应用层命令协议 v1.1

## 1. 概述
本文档定义 Lyre MIDI 控制器与上位机（Web App）之间的应用层命令格式。命令基于 Jigsaw MIDI Transport 协议提供的透明字节流传输，采用定界帧内分号分隔的键值对明文编码。

### 1.1. 设计目标
- **极简解析**：无需任何解析库，MCU 侧可用少量固定缓冲完成解码。
- **人类可读**：调试与日志记录直观，便于开发与现场排错。
- **紧凑高效**：避免 JSON 的结构冗余，适合低带宽 MIDI 传输。
- **可扩展**：支持部分字段更新，新增参数不会破坏老版本兼容性。

### 1.2. 传输层依赖
- 传输层由《Jigsaw MIDI Transport 协议》提供可靠的双向字节流。应用层帧直接通过该字节流发送，无需额外封装。

## 2. 帧格式

### 2.1. 定界与结构
- 每一帧以 **`[`** (ASCII 0x5B) 开始，以 **`]`** (ASCII 0x5D) 结束。
- 帧内包含一个或多个 **`key=value`** 对，各对之间用 **`;`** (ASCII 0x3B) 分隔。
- **首个字段必须为 `cmd=<命令名>`**。
- 字段顺序除 `cmd` 必须为首位外，其余不限。
- 帧内**不允许**出现多余的空格、换行或任何不可见字符。

**合法示例**：
```
[cmd=handshake;]
[cmd=handshake_ack;id=LYRE-A1B2;ver=1.0;proto_ver=1.1]
[cmd=save_config;0_midi_ch=1;0_cc=70;0_min=100;0_max=4000;1_cc=71;3_max=4320]
```

### 2.2. 字符集与转义
- **允许字符**：大小写字母、数字、下划线 `_`、连字符 `-`、点 `.`、冒号 `:`、逗号 `,`、管道符 `|`。  
  键和值中**禁止**出现 `[`、`]`、`;`、`=` 这四个结构字符。
- 设备 ID 等字符串由固件保证不包含上述禁止字符。上位机拼接数值时禁止在值中插入非法字符。
- **无转义机制**：协议不提供字符转义，以保持极简。所有复杂数据通过子分隔符（如 `,`）组合，确保始终在合法字符集内。

### 2.3. 帧长度限制
- 帧数据部分（`[`...`]`之间）**不应超过 256 字节**。实测最大配置帧（全 4 推杆 × 4 属性）约 200 字节，MCU 接收缓冲区 512 字节，余量充足。
- 上位机无需主动计算长度，但应避免在单帧内填充超大字段（如长字符串描述）。

### 2.4. 数值约定
- 所有数值使用十进制 ASCII 字符串表示，不加前导零（除非值为 0，用 `0` 表示）。
- MIDI 通道在应用层统一使用 **1‑based**（即用户看到的通道 1‑16），设备内部自动转换为 0‑based。
- ADC 原始值为 0‑4095 的整数。

## 3. 命令集

### 3.1. 命名规范
- `pot`：推杆索引，取值 0‑3，表示 4 个推杆之一。
- `midi_ch`：MIDI 通道，取值 1‑16。
- `cc`：MIDI CC 号，取值 0‑127。
- `min` / `max`：ADC 校准最小/最大值。

### 3.2. 握手与设备识别
| 命令 | 方向 | 格式 | 说明 |
| :--- | :--- | :--- | :--- |
| handshake | 上位机→设备 | `[cmd=handshake;]` | 发起应用层握手。 |
| handshake_ack | 设备→上位机 | `[cmd=handshake_ack;id=<设备ID>;ver=<固件版本>;proto_ver=<协议版本>]` | 回应握手，提供身份信息。 |

**字段说明**：
- `id`：设备唯一标识，格式 `LYRE-<4字符大写十六进制>`，如 `LYRE-A1B2`。
- `ver`：Lyre 固件主版本号。
- `proto_ver`：本应用层协议版本号（当前 `1.1`）。

### 3.3. 读取配置
| 命令 | 方向 | 格式 | 说明 |
| :--- | :--- | :--- | :--- |
| read_config | 上位机→设备 | `[cmd=read_config;]` | 请求读取设备永久存储的当前配置。 |
| read_config | 设备→上位机 | `[cmd=read_config;result=0;<pot>_midi_ch=...;<pot>_cc=...;...]` | 返回配置。若设备无已保存配置，返回出厂默认值。 |

**配置字段格式**：
- 每个推杆使用编号前缀 `0_`、`1_`、`2_`、`3_`，后接属性名：
  - `<pot>_midi_ch`：MIDI 通道
  - `<pot>_cc`：CC 号
  - `<pot>_min`：校准最小值
  - `<pot>_max`：校准最大值
- 必须返回 4 个推杆的完整配置，每个推杆至少包含上述 4 个字段。

**示例**：
```
上位机 → 设备: [cmd=read_config;]
设备 → 上位机: [cmd=read_config;result=0;0_midi_ch=1;0_cc=70;0_min=100;0_max=4000;1_midi_ch=1;1_cc=71;1_min=100;1_max=4000;2_midi_ch=1;2_cc=72;2_min=100;2_max=4000;3_midi_ch=1;3_cc=73;3_min=100;3_max=4000]
```

### 3.4. 获取 ADC 原始值（用于校准）
| 命令 | 方向 | 格式 | 说明 |
| :--- | :--- | :--- | :--- |
| get_adc | 上位机→设备 | `[cmd=get_adc;pot=<推杆>]` | 请求指定推杆的当前 ADC 原始值。 |
| adc_value | 设备→上位机 | `[cmd=adc_value;pot=<推杆>;raw=<ADC值>]` | 返回 ADC 原始值。 |

- `pot`：0‑3。
- `raw`：当前 ADC 采样值 0‑4095。

**示例**：
```
上位机 → 设备: [cmd=get_adc;pot=0]
设备 → 上位机: [cmd=adc_value;pot=0;raw=110]
```

### 3.5. 保存配置
| 命令 | 方向 | 格式 | 说明 |
| :--- | :--- | :--- | :--- |
| save_config | 上位机→设备 | `[cmd=save_config;<字段列表>]` | 保存配置字段到设备永久存储。支持全量或部分字段。 |
| save_config_ack | 设备→上位机 | `[cmd=save_config_ack;status=ok]` | 保存成功确认。 |
| error | 设备→上位机 | `[cmd=error;code=<错误码>;msg=<描述>]` | 保存失败时返回错误帧。 |

**部分更新语义**：
- 可以只发送需要修改的字段，例如用户只更改了推杆 2 的 CC 号，可仅发送 `[cmd=save_config;2_cc=72]`。设备只更新该字段，其他字段保持不变。
- 也可以全量发送 4 个推杆的所有字段，与 `read_config` 应答格式一致。
- 字段名必须符合 `<pot>_<属性>` 规范，参数值合法。

**示例（全量）**：
```
上位机 → 设备: [cmd=save_config;0_midi_ch=1;0_cc=70;0_min=95;0_max=4010;1_midi_ch=1;1_cc=71;1_min=100;1_max=4000;2_midi_ch=1;2_cc=72;2_min=100;2_max=4000;3_midi_ch=1;3_cc=73;3_min=100;3_max=4000]
设备 → 上位机: [cmd=save_config_ack;status=ok]
```

**示例（部分）**：
```
上位机 → 设备: [cmd=save_config;2_cc=72;2_min=80;2_max=4020]
设备 → 上位机: [cmd=save_config_ack;status=ok]
```

### 3.6. 临时校准
| 命令 | 方向 | 格式 | 说明 |
| :--- | :--- | :--- | :--- |
| calibrate | 上位机→设备 | `[cmd=calibrate;pot=<推杆>;min=<最小值>;max=<最大值>]` | 设置指定推杆的校准值，**仅写入内存，即时生效，断电丢失**。 |
| calibrate_ack | 设备→上位机 | `[cmd=calibrate_ack;result=0]` | 校准值成功应用。 |
| error | 设备→上位机 | `[cmd=error;code=<错误码>;msg=<描述>]` | 参数非法或通信错误。 |

- `pot`：0‑3。
- `min`、`max`：必须满足 `min < max`，否则设备应返回错误。
- 临时校准后，上位机应提示用户执行 `save_config` 以持久化。

**示例**：
```
上位机 → 设备: [cmd=calibrate;pot=0;min=95;max=4010]
设备 → 上位机: [cmd=calibrate_ack;result=0]
上位机 → 设备: [cmd=save_config;0_min=95;0_max=4010]   (部分保存)
设备 → 上位机: [cmd=save_config_ack;status=ok]
```

### 3.7. 恢复出厂设置
| 命令 | 方向 | 格式 | 说明 |
| :--- | :--- | :--- | :--- |
| factory_reset | 上位机→设备 | `[cmd=factory_reset;]` | 清除所有用户配置，恢复出厂默认值并写入永久存储。 |
| factory_reset_ack | 设备→上位机 | `[cmd=factory_reset_ack;status=ok]` | 恢复成功确认。 |

设备在执行完成后自动重新加载默认配置，上位机应在收到 `ack` 后延迟约 500ms 再重新读取配置。

### 3.8. 错误应答
当设备无法解析或执行命令时，返回错误帧：
```
[cmd=error;code=<错误码>;msg=<简短描述>]
```
**错误码定义**：

| 代码 | 含义 | 示例 msg |
| :--- | :--- | :--- |
| 1 | 帧格式错误 | `"missing cmd field"` |
| 2 | 未知命令 | `"unknown command: xxx"` |
| 3 | 参数缺失或非法 | `"missing 'pot' parameter"`, `"min must be less than max"` |
| 4 | 内部执行失败 | `"flash write error"` |
| 5 | 推杆索引越界 | `"pot must be 0-3"` |

## 4. 交互流程示例

### 4.1. 启动与配置读取
```
上位机 → 设备: [cmd=handshake;]
设备 → 上位机: [cmd=handshake_ack;id=LYRE-A1B2;ver=1.0;proto_ver=1.1]
上位机 → 设备: [cmd=read_config;]
设备 → 上位机: [cmd=read_config;result=0;0_midi_ch=1;0_cc=70;0_min=100;0_max=4000;1_midi_ch=1;1_cc=71;1_min=100;1_max=4000;2_midi_ch=1;2_cc=72;2_min=100;2_max=4000;3_midi_ch=1;3_cc=73;3_min=100;3_max=4000]
```

### 4.2. 校准与部分保存
```
上位机 → 设备: [cmd=get_adc;pot=0]
设备 → 上位机: [cmd=adc_value;pot=0;raw=110]
(多次轮询，获取 min=95, max=4010)
上位机 → 设备: [cmd=calibrate;pot=0;min=95;max=4010]
设备 → 上位机: [cmd=calibrate_ack;result=0]
上位机 → 设备: [cmd=save_config;0_min=95;0_max=4010]
设备 → 上位机: [cmd=save_config_ack;status=ok]
```

### 4.3. 错误处理
```
上位机 → 设备: [cmd=get_adc;pot=4]
设备 → 上位机: [cmd=error;code=5;msg=pot must be 0-3]
上位机 → 设备: [cmd=calibrate;pot=0;min=1000;max=800]
设备 → 上位机: [cmd=error;code=3;msg=min must be less than max]
```

## 5. 实现注意事项

### 5.1. MCU 解析建议
- 使用 512 字节接收缓冲区，完全足够容纳任何合法帧。
- 遇到 `[` 开始记录，后续字符直至 `]` 作为帧内容。
- 解析时顺序扫描，按 `;` 分割为字段，对每个字段查找 `=` 分离键值。
- 键值对可存入固定大小的数组（如 20 个条目），根据 `cmd` 值调用处理函数。
- **严禁动态内存分配**。

### 5.2. 上位机构造要求
- 确保帧中无非法字符（`[`、`]`、`;`、`=`）。数值字段天然安全。
- 支持部分字段更新，只发送变化了的配置参数，减少传输量。
- 收到错误帧时，向用户提供清晰提示并根据错误码采取合理恢复动作。

### 5.3. 版本兼容
- 协议版本 `proto_ver` 采用 `主版本.次版本`。主版本变更表示不兼容修改，次版本增加向后兼容的扩展字段。上位机应检查版本，必要时提示用户更新。
