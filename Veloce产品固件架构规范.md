# 嵌入式固件架构设计规范：信息管线星型架构 (v1.1)

> **文档定位**：本文档是所有基于“信息管线星型架构”的嵌入式固件项目的**通用元规则（Meta-Rules）**。
> 任何具体产品（如 Lyre、Veloce 等）在编写其产品级架构文档（如 `LYRE_ARCHITECTURE.md`）时，必须严格遵循本规范，并在此基础上实例化具体的业务流和时序参数。

## 1. 架构愿景与核心原则
本架构旨在为多产品线的嵌入式固件提供极致的解耦与复用能力。核心目标是实现 **“乐高化”开发**：将系统拆分为高度内聚、绝对隔离的“管线（Pipeline）”，通过标准的“市场（Marketplace）”接口进行协作。

**三大铁律（不可违背）：**
1. **管线间绝对隔离**：任何管线的内部实现（包括 HAL 和 CORE）绝不允许直接调用其他管线。
2. **市场是唯一交汇点**：管线之间的信息交换，必须且只能通过各自 APP 层暴露在“市场”中的 API 进行。
3. **CORE 层零外部依赖**：CORE 层只包含跨产品通用的领域能力，绝对不依赖任何外部管线或产品特有业务。

---

## 2. 管线内部三层模型 (The 3-Tier Pipeline Model)
每条管线在物理和逻辑上必须严格划分为三层。依赖方向必须自上而下（APP -> CORE -> HAL），严禁反向依赖。

### 2.1 HAL 层 (Hardware Abstraction Layer)
* **职责**：直接操作硬件寄存器，提供最底层的读写能力（如 ADC 采样、Flash 扇区擦写、UART/USB 字节收发）。
* **特性**：与具体 MCU 型号绑定，与业务逻辑完全无关。

### 2.2 CORE 层 (Core Domain Logic)
* **职责**：提供**跨产品可复用的领域通用能力**（如数字滤波算法、协议帧解析引擎、Flash 磨损均衡算法）。
* **特性（免检产品）**：
  * **绝对不对外**：不调用任何其他管线的 API。
  * **无产品认知**：不知道当前产品的具体硬件配置（如推杆数量）。
  * **高度复用**：开发新产品时，CORE 层代码直接拷贝，一行不改。
  * **依赖倒置**：若需外部数据（如业务校验规则），必须由 APP 层通过**函数指针（回调/依赖注入）** 传入。

### 2.3 APP 层 (Application & Business Logic)
* **职责**：处理**特定产品的业务逻辑**，并作为该管线在“市场”中的唯一代表。
* **特性（乐高积木）**：
  * **对外提供服务**：实现并暴露 `_api.h` 接口，供其他管线调用。
  * **调用市场服务**：根据业务需要，调用其他管线的 API。
  * **产品定制**：开发新产品时，只需重写 APP 层。

---

## 3. “市场”机制 (The Marketplace)
“市场”在物理形态上是一个**独立的头文件目录（`market/`）**，在逻辑上代表所有管线对外暴露的 API 集合。

### 3.1 物理结构
```text
firmware/
├── pipelines/          # 管线的内部实现（私有，对外不可见）
│   ├── potentiometer/  # (pot_hal.c, pot_core.c, pot_app.c, 内部.h)
│   ├── cmd_config/     # (cmd_core.c, cmd_app.c, config_manager.c, 内部.h)
│   └── storage/        # (storage_hal.c, storage_core.c, storage_app.c, 内部.h)
│
├── market/             # 市场（公共 API，全局可见）
│   ├── pot_api.h       # 电位器管线的“招牌”
│   ├── cmd_cfg_api.h   # 命令与配置管线的“招牌”
│   └── storage_api.h   # 存储管线的“招牌”
│
└── main.c              # 主循环，负责初始化和驱动注入
```

### 3.2 调用规则
* **管线内部代码**（`pipelines/` 下的 `.c` 文件）：
  * ✅ 可以 `#include` 自己管线内部的 `.h` 文件。
  * ✅ 可以 `#include "market/xxx_api.h"`（调用其他管线）。
  * ❌ 绝对不能 `#include` 其他管线的内部 `.h` 文件（如 `../cmd_config/cmd_core.h`）。

## 4. 关键管线架构定义 (通用抽象)

### 4.1 命令与配置管线 (Command & Config Pipeline)
* **CORE 层 (`base_proto`)**：通用协议引擎。负责字节流拼接、基础命令（握手/心跳）、帧解析、TX 调度。通过命令表（Command Table）将未知命令委托给 APP 层。
* **APP 层**：实现产品特有命令的 Handler。在 RAM 中持有当前设备的**配置快照 (Single Source of Truth)**，提供线程安全的查询 API 给其他管线。

### 4.2 电位器管线 (Potentiometer Pipeline)
* **CORE 层**：信号处理算法（滑动平均滤波、死区计算、迟滞处理）。
* **APP 层**：获取 CORE 层处理后的 Raw 值，通过“市场”获取映射规则，计算出最终输出值（如 MIDI CC）。

### 4.3 存储管线 (Storage Pipeline)
* **CORE 层**：Flash 驱动封装、磨损均衡、CRC 校验。
* **APP 层**：提供面向业务的读写接口（如 `storage_save_config()`），屏蔽底层 Flash 地址和扇区细节。

## 5. 并发与数据一致性规范
在 RTOS 或裸机中断环境下，跨管线数据交换必须保证一致性。

1. **原子读取**：配置管线的 APP 层在提供查询 API 时，若配置结构体较大，必须在临界区（关中断或互斥锁）内完成数据拷贝，**禁止返回指向内部 RAM 的指针**。
2. **快照切换**：对于高频读取的数据（如映射表），采用“双缓冲/指针原子切换”机制。写入新配置时在后台 Buffer 准备，完毕后原子切换指针，避免读取到脏数据。

## 6. 驱动模型与时序规范 (Drive Model & Timing)
本章节定义系统“动力”的注入与传递元规则。具体产品的驱动时序需在产品级文档中实例化。

### 6.1 驱动源分类 (Drive Source Taxonomy)
系统的所有“动力”必须且只能来自以下三种驱动源：

| 驱动源类型 | 触发机制 | 典型场景 | 进入系统的入口 |
|---|---|---|---|
| **周期性驱动** | 硬件定时器中断或 RTOS 任务 | 传感器采样、心跳包 | 主循环或定时器回调 |
| **事件驱动** | 外部硬中断或异步信号 | 收到 USB 数据、按键按下 | 中断服务程序 (ISR) |
| **级联驱动** | 上游管线处理完成后主动调用下游 | 计算完数值后调用发送接口 | 管线 APP 层内部调用 |

### 6.2 驱动传递原则
1. **单一入口，单向流动**：每条业务流必须有且只有一个驱动入口。驱动力只能从上游向下游传递，**禁止形成驱动环路**。
2. **同步 vs 异步边界**：同一条业务流内，若上游同步调用下游 API，必须在产品文档中声明该 API 的**最大执行时间**。耗时操作（如 Flash 写入）必须通过异步队列解耦。
3. **ISR 边界铁律**：ISR（中断服务程序）**只负责“收集信号”**。绝不允许在 ISR 中直接调用市场 API。信号必须通过队列或标志位传递到主循环，由主循环统一调度。

### 6.3 业务流声明规范
在产品级架构文档中，必须为每一条业务流绘制**驱动流图**，并声明以下要素：
* **驱动源**：类型与触发频率/条件。
* **驱动路径**：起点 → 中间节点 → 终点（标明同步/异步调用）。
* **时序约束**：端到端最大延迟、各环节耗时预算。
* **失败处理**：超时、重试、丢弃或告警策略。

## 7. 工程化保障手段
为防止架构在长期迭代中腐化，必须采用以下工程化手段强制隔离：

1. **编译期物理隔离**：利用构建系统（如 CMake 的 `target_include_directories`），严格限制各管线 target 的头文件搜索路径，使违规的 `#include` 在编译期直接报错。
2. **API 契约注释**：每个 `market/xxx_api.h` 头部必须声明 `@consumers`（谁调用我）和 `@dependencies`（我内部依赖谁）。
3. **依赖图静态分析**：提供脚本（如 `tools/gen_market_deps.py`），定期扫描 `#include "market/..."` 生成依赖关系图，用于代码审查和重构评估。

---

## 8. 新产品开发 SOP（乐高拼装指南）

当启动一个新产品（例如：无 Flash、8 推杆的简易控制器 *Veloce-Mini*）时：

1. **复用 CORE 层（直接拷贝）**：拷贝 `base_proto`、滤波算法等。移除不需要的管线（如存储管线）。
2. **定制 APP 层（重新编写）**：重写命令 Handler（去掉写 Flash 逻辑），修改映射逻辑（使用硬编码默认值）。
3. **配置市场（更新 API）**：更新 `market/` 目录下的接口定义，移除不需要的 API。
4. **定义业务流（编写产品文档）**：基于本规范第 6.3 节，编写该产品的专属业务流驱动图。

# 附录：产品级架构设计文档模板
> **使用说明**：以下为具体产品（以 Lyre 为例）的架构设计文档示例。在实际开发中，应将其保存为独立文件（如 `docs/LYRE_ARCHITECTURE.md`）。

## A. 产品概述
* **产品名称**：Lyre 4-推杆 MIDI 控制器
* **硬件基础**：STM32G4, 4x 10k 电位器, 16MB SPI Flash, USB-C
* **遵循规范**：《信息管线星型架构 v1.1》

## B. 市场 API 清单 (Lyre 实例化)
* `market/pot_api.h`: `pot_get_raw()`, `pot_get_midi()`
* `market/cmd_cfg_api.h`: `config_get_pot_mapping()`, `config_get_snapshot()`
* `market/storage_api.h`: `storage_save_config()`, `storage_load_config()`
* `market/led_api.h`: `led_pulse_activity()`

## C. 核心业务流定义 (Drive Flows)

### 业务流 1：电位器采样与 MIDI 发送
* **驱动源**：周期性驱动，主循环每 50ms 触发一次 (20Hz)
* **驱动路径**：
  ```text
  主循环 (50ms Tick)
    ▼
  电位器 APP (pot_app_poll)
    ├── [同步] 电位器 CORE：4路 ADC 读取与滑动平均滤波
    ├── [同步] 命令配置 API：读取 4路 Min/Max/CC 映射 (config_get_pot_mapping)
    ├── [本地] Raw 值映射为 MIDI CC (0-127)
    ├── [同步] MIDI 引擎 API：发送 CC 消息 (midi_send_cc)
    └── [同步] LED API：触发活动指示灯闪烁 (led_pulse_activity)
  ```
* **时序约束**：端到端 ≤ 5ms。配置查询必须从 RAM 快照读取（≤ 0.5ms）。
* **失败处理**：ADC 失败则跳过本周期；配置无效则使用默认映射 (0-127)；MIDI 队列满则丢弃不阻塞。

### 业务流 2：上位机写入配置
* **驱动源**：事件驱动，USB 收到 `CMD_WRITE_CFG` 命令帧
* **驱动路径**：
  ```text
  USB 中断 (ISR) -> [队列] -> 主循环
    ▼
  命令配置 APP (cmd_app_process_rx)
    ├── [同步] 命令 CORE (base_proto)：帧解析
    ├── [本地] 校验 16 个字段合法性
    ├── [异步] 存储 API：写入 Flash (storage_save_config) -> 触发完成回调
    ├── [本地] 更新 RAM 配置快照 (双缓冲原子切换)
    └── [同步] 命令 CORE：组装并发送 ACK 响应
  ```
* **时序约束**：端到端 ≤ 50ms。Flash 写入在后台任务执行，不阻塞 ACK 发送。
* **失败处理**：校验失败返回 NACK；Flash 写入失败返回 NACK 且 RAM 快照回滚。
