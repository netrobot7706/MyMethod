# 嵌入式固件架构设计规范：信息管线星型架构 (v1.0)

## 1. 架构愿景与核心原则
本架构旨在为多产品线（如 MIDI 控制器系列）的嵌入式固件提供极致的解耦能力。核心目标是实现 **“乐高化”开发**：将系统拆分为高度内聚、绝对隔离的“管线（Pipeline）”，通过标准的“市场（Marketplace）”接口进行协作。

**三大铁律（不可违背）：**
1. **管线间绝对隔离**：任何管线的内部实现（包括 HAL 和 CORE）绝不允许直接调用其他管线。
2. **市场是唯一交汇点**：管线之间的信息交换，必须且只能通过各自 APP 层暴露在“市场”中的 API 进行。
3. **CORE 层零外部依赖**：CORE 层只包含跨产品通用的领域能力，绝对不依赖任何外部管线或产品特有业务。

---

## 2. 管线内部三层模型 (The 3-Tier Pipeline Model)
每条管线在物理和逻辑上必须严格划分为三层。依赖方向必须自上而下（APP -> CORE -> HAL），严禁反向依赖。

### 2.1 HAL 层 (Hardware Abstraction Layer)
* **职责**：直接操作硬件寄存器，提供最底层的读写能力（如 ADC 采样、Flash 扇区擦写、UART 字节收发）。
* **特性**：与具体 MCU 型号绑定，与业务逻辑完全无关。

### 2.2 CORE 层 (Core Domain Logic)
* **职责**：提供**跨产品可复用的领域通用能力**（如数字滤波算法、协议帧解析引擎、Flash 磨损均衡算法）。
* **特性（免检产品）**：
  * **绝对不对外**：不调用任何其他管线的 API。
  * **无产品认知**：不知道当前产品有几个推杆、什么型号。
  * **高度复用**：开发新产品时，CORE 层代码直接拷贝，一行不改。
  * **依赖倒置**：如果 CORE 层需要外部数据（如业务校验规则），必须由 APP 层通过**函数指针（回调/依赖注入）** 传入。

### 2.3 APP 层 (Application & Business Logic)
* **职责**：处理**特定产品（如 Lyre）的业务逻辑**，并作为该管线在“市场”中的唯一代表。
* **特性（乐高积木）**：
  * **对外提供服务**：实现并暴露 `_api.h` 接口，供其他管线调用。
  * **调用市场服务**：根据业务需要，调用其他管线的 API。
  * **产品定制**：开发新产品时，只需重写 APP 层。

---

## 3. “市场”机制 (The Marketplace)
“市场”不是一个具体的代码实体，而是一个**逻辑概念**，代表所有管线对外暴露的 API 集合。

* **接口隔离**：每个管线对外提供一个独立的头文件（如 `pot_pipeline_api.h`），其他管线只 `#include` 自己需要的头文件。
* **信息流向**：
  * **请求方 APP** -> 调用目标管线 API -> **目标方 APP** -> 调用目标方 CORE -> 调用目标方 HAL。
  * 请求方的 CORE 永远不参与跨管线调用。

---

## 4. 关键管线架构定义

### 4.1 命令与配置管线 (Command & Config Pipeline)
负责设备的对外通信（协议解析）与全局状态（配置）管理。

* **HAL 层**：无（或复用底层 UART/USB 驱动）。
* **CORE 层 (`base_proto` 基础协议库)**：
  * **职责**：通用协议引擎。负责字节流拼接（Jigsaw）、基础命令（握手、心跳）、帧解析、TX 调度。
  * **边界**：完全不知道 Lyre 的特有命令（如 `write_cfg`）是什么。通过命令表（Command Table）和回调机制，将未知命令委托给 APP 层。
* **APP 层 (业务逻辑与配置中心)**：
  * **Handler 函数**：产品特有命令的处理逻辑。负责校验字段、拼装数据结构，并**通过市场**调用其他管线（如调用存储管线保存配置，调用电位器管线读取当前状态）。
  * **配置管理中心 (Config Manager)**：在 RAM 中持有当前设备的配置快照（Single Source of Truth），提供线程安全的查询 API 给其他管线（如 `config_get_pot_mapping()`）。

### 4.2 电位器管线 (Potentiometer Pipeline)
负责模拟输入信号的采集、处理与映射。

* **HAL 层**：ADC 寄存器配置与原始值读取。
* **CORE 层**：信号处理算法（如滑动平均滤波、死区计算、迟滞处理）。
* **APP 层**：
  * **映射计算**：获取 CORE 层处理后的 Raw 值，通过“市场”调用**命令与配置管线**的 API 获取 Min/Max/CC 映射规则，计算出最终的 MIDI 值。
  * **对外 API**：提供 `pot_get_raw()`, `pot_get_midi()` 等接口。

### 4.3 存储管线 (Storage Pipeline)
负责非易失性数据的持久化。

* **HAL 层**：Flash 扇区擦除、页编程、读取。
* **CORE 层**：Flash 驱动封装、简单的磨损均衡或校验和（CRC）计算。
* **APP 层**：提供面向业务的读写接口（如 `storage_save_config()`, `storage_read_calibration()`），屏蔽底层 Flash 地址和扇区细节。

---

## 5. 并发与数据一致性规范
在 RTOS 或裸机中断环境下，跨管线数据交换必须保证一致性。

1. **原子读取**：配置管线的 APP 层在提供查询 API 时，若配置结构体较大，必须在临界区（如关中断或互斥锁）内完成数据拷贝，禁止返回指向内部 RAM 的指针。
2. **快照切换**：对于高频读取的数据（如 MIDI 映射表），采用“双缓冲/指针原子切换”机制。写入新配置时在后台 Buffer 准备，准备完毕后原子切换指针，避免读取到“写了一半”的脏数据。

---

## 6. 新产品开发 SOP（乐高拼装指南）

当启动一个新产品（例如：无 Flash、8 推杆的简易 MIDI 控制器 *Veloce-Mini*）时，开发流程如下：

1. **复用 CORE 层（直接拷贝）**：
   * 拷贝 `base_proto` (命令 CORE)。
   * 拷贝 `pot_core` (滤波算法)。
   * 移除 `storage_pipeline` (因为不需要 Flash)。
2. **定制 APP 层（重新编写）**：
   * **命令 APP**：重写 Handler，去掉 `write_cfg` 等需要 Flash 的命令，修改校验逻辑以适配 8 推杆。
   * **电位器 APP**：修改映射逻辑（例如：不再调用配置管线查 Min/Max，而是使用硬编码的默认映射）。
3. **配置市场（更新 API）**：
   * 更新 `market` 目录下的接口定义，移除不需要的 API，确保编译通过。
