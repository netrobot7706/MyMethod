# 📑 18位自包含无损配置传输协议规范 (Pitch Bend 方案)
## 1. 概述 & 设计哲学
### 1.1 目的
本协议旨在解决在标准 MIDI 物理总线或虚拟 MIDI 链路上，如何安全、无损、抗干扰地传输自定义文本/二进制配置命令（形如 [cmd=...;]）的问题。
### 1.2 核心设计哲学
 * **原子性与自包含性 (Atomicity & Self-containment)**：每一个下发的 MIDI 消息都是一个独立的、自解释的宇宙。数据、序号、状态三合一，杜绝跨包状态依赖。
 * **零拷贝与无状态重组 (Zero-copy & Out-of-order Reassembly)**：接收端无需维护高风险的动态读写指针，利用绝对序号直接映射缓冲区数组下标，天生免疫网络时序抖动和乱序。
 * **分层彻底解耦 (Layer Decoupling)**：本协议属于**传输层协议**，其唯一职责是将底层的离散 MIDI 消息完美还原为连续的字节流，对应用层文本的特定语义（如 cmd=）保持完全透明。
## 2. 物理层位映射 (Bit Mapping)
协议强行打通并压榨了标准 MIDI 弯音消息 (Pitch Bend, 状态码 0xEx) 的全部 **18位 (Bits)** 物理资产，其在 3 个 MIDI 字节中的分布如下表所示：
| 字节序号 | 字节流定义 (Binary) | 包含的协议字段 | 详细说明 |
|---|---|---|---|
| **Byte 1** | 1110 [F1][F0][S7][S6] | 状态头 + 帧控制(2bit) + 序号高2位 | 1110 为 Pitch Bend 固定头；低4位（原通道位）被征用。 |
| **Byte 2** | 0[P6][P5][P4][P3][P2][P1][P0] | 属性标志位 0 + 载荷低7位 | 标准 MIDI 数据字节约束，最高位强制为 0。 |
| **Byte 3** | 0[S5][S4][S3][S2][S1][S0][P7] | 属性标志位 0 + 序号低6位 + 载荷最高1位 | 标准 MIDI 数据字节约束，最高位强制为 0。 |
> 📌 **字段符号说明**：[F] = 帧控制位 (Frame Control), [S] = 序列号位 (Sequence Number), [P] = 数据载荷位 (Payload)。
> 
## 3. 协议字段详解
通过位移复合运算，这 18 位资产被完美重组为以下三个逻辑字段：
```
+-------------------+-----------------------------------+-----------------------------------+
|  Frame Control    |        Sequence Number            |             Payload               |
|    (bit 17-16)    |          (bit 15-8)               |            (bit 7-0)              |
+-------------------+-----------------------------------+-----------------------------------+

```
### 3.1 帧控制 (Frame Control) —— 2 Bits
用于控制状态机的宏观跳转，并引入**会话轮替 (Session Alternating)** 机制区分连续指令：
 * 00 (Session A Data)：当前字节属于“会话 A”的数据帧，同时若 Sequence Number == 0，则兼任“会话 A 开始帧”。
 * 01 (Session B Data)：当前字节属于“会话 B”的数据帧，同时若 Sequence Number == 0，则兼任“会话 B 开始帧”。
 * 10 (Frame End / EOF)：宣告当前命令帧传输结束，触发接收端应用层解析器。
 * 11 (System Abort / Reset)：系统级强制复位信号。接收端收到后必须立刻清空当前接收缓冲区，中止一切解析流程。
### 3.2 序列号 (Sequence Number) —— 8 Bits
 * **范围**：0x00 至 0xFF (0 - 255)。
 * **核心功用**：充当数据载荷在接收缓冲区（Rx_Buffer）中的**绝对数组下标 (Index)**。由于单条文本指令长度极少超过 256 字节，该序号在同一次会话中具有唯一性。
### 3.3 数据载荷 (Payload) —— 8 Bits
 * **范围**：0x00 至 0xFF。
 * **物理含义**：还原后的 1 字节纯文本（如字符 [、c、m、d、; 等）或二进制数据。
## 4. 端到端核心工作流与状态机
### 4.1 发送端 (Web UI / 上位机) 行为准则
 1. **打包准备**：将待发送的命令字符串（如 [cmd=set;param=1;]）拆分为字符数组。
 2. **会话选择**：检查上一次发送使用的帧控制标志。如果上次是 Session A (00)，本次必须轮替为 Session B (01)。
 3. **逐包组装**：
   * 遍历字符串，当前字符的索引即为 Sequence Number，字符 ASCII 码即为 Payload。
   * 严格按照 **第 2 节** 的位映射表，利用位运算将 FC、SEQ、Payload 揉进 3 个 MIDI 字节。
 4. **尾包发送**：字符串发送完毕后，必须额外追加发送一个 FC = 10 (Frame End) 的终止包，该包的 SEQ 填写当前帧的总长度，Payload 填 0x00。
### 4.2 接收端 (嵌入式 MCU / 固件) 状态机逻辑
#### 🛠️ 前置硬性硬件配置
 * **混杂模式 (MIDI Omni Mode)**：MCU 固件驱动层必须关闭单通道过滤，**必须接收所有通道 (0-15)** 的 Pitch Bend 消息。
#### 🔄 核心状态机算法
```
               [ 收到任意 Pitch Bend 消息 ]
                            │
              (位运算合成为 18位 虚拟寄存器 V)
                            │
              ┌─────────────┴─────────────┐
              ▼                           ▼
       [ FC == 11 (Abort) ]       [ FC == 10 (End) ]
              │                           │
      (清空 Rx_Buffer)            (校验数据完整性)
      (重置 Session)                      │
              │                 ┌─────────┴─────────┐
              ▼                 ▼                   ▼
            [退出]           [通过]              [失败]
                                │                   │
                         (交付应用层解析)       (抛弃该帧)
                                │                   │
                                └─────────┬─────────┘
                                          ▼
                                        [退出]

```
 1. **解析中断入口**：收到 Pitch Bend 消息，当场提取字段：
   ```c
   uint8_t fc   = (byte1 & 0x0C) >> 2;
   uint8_t seq  = ((byte1 & 0x03) << 6) | ((byte3 & 0x7E) >> 1);
   uint8_t play = ((byte3 & 0x01) << 7) | (byte2 & 0x7F);
   
   ```
 2. **分支预测判断**：
   * **若 fc == 11 (Abort)**：清空缓冲区，复位接收状态，退出。
   * **若 fc == 10 (End)**：读取此时的 seq（即发送端宣告的总长度）。检查本地缓冲区 0 到 seq-1 位置是否都有有效数据注入。若校验通过，**触发回调函数/推入队列**，将数据流交付给应用层（文本解析器）；若有空洞，直接报错丢弃。
   * **若 fc == 00 或 01 (Data)**：
     * **新旧会话判定**：如果本地记录的当前会话 ID 与收到的 fc 不同，且 seq == 0，说明新一轮调参指令开始，立刻隐式刷新缓冲区，更新当前会话 ID。
     * **绝对映射填空**：无视当前接收指针，直接执行：Rx_Buffer[seq] = play; 并将该位置的“有效标志位”置 1。
## 5. 协议防御能力评估 (Security & Robustness)
 * **抗乱序能力**：完美。由于 Rx_Buffer[seq] = play 的存在，即便 Byte 50 比 Byte 49 先到达 MCU，也会精准坐在自己的位置上，不影响最终整体的完整性。
 * **抗断线/丢包能力**：极强。若中间丢失任意一包，在收到 FC == 10 (End) 时，接收端扫描 Rx_Buffer 会暴露出未被填补的空洞，状态机将立刻熔断，拒绝将致盲的畸形命令交付给应用层，从而 100% 保护了应用层内存的安全。
