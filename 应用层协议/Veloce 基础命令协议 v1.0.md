# Veloce 基础命令协议 v1.0

## 1. 概述

本文档定义 Veloce 系列 MIDI 控制器产品的**通用基础命令协议**。它规定了应用层帧格式、设备状态机、握手与恢复出厂等所有产品共用的机制。

基础协议基于 Jigsaw MIDI Transport 协议提供的透明字节流传输，采用定界帧内分号分隔的键值对明文编码。具体产品的业务命令（如配置读写、数据采集等）由各产品的《产品命令协议》定义。

### 1.1. 设计目标
- **极简解析**：无需任何解析库，MCU 侧可用少量固定缓冲完成解码。
- **人类可读**：调试与日志记录直观，便于开发与现场排错。
- **紧凑高效**：避免 JSON 的结构冗余，适合低带宽 MIDI 传输。
- **可扩展**：新增参数不破坏老版本兼容性。
- **健壮**：具备帧同步恢复、超时处理，适应真实硬件环境。
- **复用**：基础协议可被所有后续产品继承，业务命令按产品定制。

### 1.2. 传输层依赖
- 传输层由《Jigsaw MIDI Transport 协议》提供可靠的双向字节流，最大帧长 8192 字节。应用层帧直接通过该字节流发送，无需额外封装。

## 2. 帧格式

### 2.1. 定界与结构
- 每一帧以 **`[`** (ASCII 0x5B) 开始，以 **`]`** (ASCII 0x5D) 结束。
- 帧内包含一个或多个 **`key=value`** 对，各对之间用 **`;`** (ASCII 0x3B) 分隔。
- **首个字段必须为 `cmd=<命令名>`**。
- 字段顺序除 `cmd` 必须为首位外，其余不限。
- **最后一个字段后不加 `;`**。帧内不允许出现连续分号。
- 帧内**不允许**出现多余的空格、换行或任何不可见字符。

**合法示例**：
```
[cmd=handshake]
[cmd=handshake_ack;id=Veloce-A1B2;ver=1.0;base_proto_ver=1.0;biz_proto_ver=1.0;max_frame_len=512]
```

### 2.2. 字符集与转义
- **允许字符**：大小写字母、数字、下划线 `_`、连字符 `-`、点 `.`、冒号 `:`、逗号 `,`、管道符 `|`。
- 键和值中**禁止**出现 `[`、`]`、`;`、`=` 这四个结构字符。
- **特别说明**：空格 (0x20) 不在允许字符集中。所有描述性文本必须使用下划线 `_` 替代空格。
- 设备 ID 等字符串由固件保证不包含上述禁止字符。上位机拼接数值时禁止在值中插入非法字符。
- **无转义机制**：协议不提供字符转义，以保持极简。

### 2.3. 帧长度限制
- **默认帧长**：基础协议定义默认最大帧数据长度（`[`...`]`之间）为 **512 字节**。
- **产品覆盖**：具体产品可在其《产品命令协议》中定义更大的帧长，上限不超过传输层最大帧长 **8192 字节**。
- **能力交换**：设备在握手时通过 `max_frame_len` 字段告知上位机本产品实际支持的最大帧长。上位机发送帧时不得超过该值。
- **溢出处理**：如果接收方在收到 `[` 后累计接收字节数超过本产品定义的 `max_frame_len` 仍未遇到 `]`，应丢弃当前帧并返回错误帧 `[cmd=error;code=1;msg=frame_too_long]`。

### 2.4. 帧同步恢复机制
接收方在解析字节流时，必须遵循以下同步恢复规则：
1. **帧开始**：遇到 `[` 时，立即重置帧缓冲区，开始接收新帧。如果此前有未完成的帧（已收到 `[` 但未收到 `]`），则无条件丢弃未完成帧。
2. **帧结束**：遇到 `]` 时，当前帧结束，将缓冲区内容提交给命令处理器，然后清空缓冲区。
3. **帧接收超时**：如果收到 `[` 后 **1000ms 内**未收到 `]`，则丢弃当前帧，清空缓冲区，返回错误帧 `[cmd=error;code=1;msg=frame_timeout]`。
4. **上位机行为**：上位机若在发送帧后检测到连接中断或页面刷新，应在重连后重新握手，并丢弃任何未完成帧。

### 2.5. 数值约定
- 所有数值使用十进制 ASCII 字符串表示，不加前导零（除非值为 0，用 `0` 表示）。
- 当前协议所有数值均为非负整数。未来扩展如需支持负数，`-` 前缀表示负号。

## 3. 设备状态机

设备在应用层维护以下状态：

| 状态 | 说明 |
|:---|:---|
| **IDLE** | 上电初始状态。设备使用已保存配置（或出厂默认值）正常工作。等待上位机握手。 |
| **CONNECTED** | 上位机完成握手后的状态。设备可接受产品协议定义的所有配置管理命令。 |
| **连接超时** | 在 CONNECTED 状态下，若 **30 秒内**未收到任何有效应用层命令，设备自动回到 IDLE 状态。下次上位机通信前需重新握手。 |

> **重要说明**：握手仅用于配置管理。设备在任何状态下都正常工作（发送 MIDI 数据等）。用户在不使用 Web App 的纯 MIDI 场景下无需握手。

### 3.1. IDLE 状态下收到配置管理命令
在 IDLE 状态下，设备收到任何需要 CONNECTED 状态的命令时，应返回错误帧：

```
[cmd=error;code=6;msg=not_connected]
```

上位机收到错误码 6 后，应自动重新握手。

### 3.2. 流模式与连接超时
若产品协议定义了流模式（设备主动持续上报数据），则在流模式激活期间，**暂停连接超时检测**。流模式关闭后恢复超时检测。流模式本身即证明连接活跃。

## 4. 通用命令

### 4.1. 命名规范
- 所有命令名采用 **`动词_名词`** 风格（如 `read_config`、`write_config`）。`handshake` 与 `factory_reset` 作为约定俗成术语保留。
- 所有命令和应答均遵循串行处理模型：**上位机应在收到当前命令的应答后再发送下一条命令**。设备不支持命令流水线。

---

### 4.2. 握手与设备识别
| 命令 | 方向 | 格式 | 说明 |
| :--- | :--- | :--- | :--- |
| handshake | 上位机→设备 | `[cmd=handshake]` | 发起应用层握手。 |
| handshake_ack | 设备→上位机 | `[cmd=handshake_ack;id=<设备ID>;ver=<固件版本>;base_proto_ver=<基础协议版本>;biz_proto_ver=<产品协议版本>;max_frame_len=<最大帧长>]` | 回应握手，提供身份、协议版本与能力信息。 |

**字段说明**：
- `id`：设备唯一标识，格式 `Veloce-<4字符大写十六进制>`，如 `Veloce-A1B2`。
- `ver`：固件主版本号（产品级别）。
- `base_proto_ver`：本基础协议版本号（当前 `1.0`）。基础协议版本变更是整个产品线的重大变更。
- `biz_proto_ver`：产品协议版本号（当前由具体产品文档定义）。上位机据此了解设备支持哪些产品特有命令。
- `max_frame_len`：本产品支持的最大帧数据长度（字节）。上位机发送任何帧时不得超过此值。取值范围：512 ~ 8192。

**版本兼容规则**：
- **基础协议**：若 `base_proto_ver` 主版本不匹配，上位机应停止通信并提示更新固件/App。次版本差异时，以较低次版本通信。
- **产品协议**：若 `biz_proto_ver` 主版本不匹配，上位机应提示更新固件/App，并在可能的情况下降级通信。次版本差异时，以较低次版本通信。

**超时**：上位机发送 `handshake` 后 **2000ms** 内未收到 `handshake_ack`，应视为超时。最多重试 3 次。

**示例**：
```
上位机 → 设备: [cmd=handshake]
设备 → 上位机: [cmd=handshake_ack;id=Veloce-A1B2;ver=1.0;base_proto_ver=1.0;biz_proto_ver=1.0;max_frame_len=512]
```

---

### 4.3. 恢复出厂设置
| 命令 | 方向 | 格式 | 说明 |
| :--- | :--- | :--- | :--- |
| factory_reset | 上位机→设备 | `[cmd=factory_reset]` | 清除所有用户配置，恢复出厂默认值并写入永久存储。 |
| factory_reset_ack | 设备→上位机 | `[cmd=factory_reset_ack;status=ok]` | 恢复成功确认。 |

**执行时序**：设备必须在 Flash 操作**完全完成**后才发送 `factory_reset_ack`。上位机收到 ACK 后应延迟至少 **500ms** 再重新握手和读取配置。

**超时**：上位机发送后 **5000ms** 内未收到应答，视为超时。不重试。

**示例**：
```
上位机 → 设备: [cmd=factory_reset]
设备 → 上位机: [cmd=factory_reset_ack;status=ok]
```

## 5. 错误应答

当设备无法解析或执行命令时，返回错误帧：

```
[cmd=error;code=<错误码>;msg=<简短描述>]
```
`msg` 使用下划线替代空格，不得包含空格字符。

### 5.1. 通用错误码（基础协议）

| 代码 | 含义 | 示例 msg |
| :--- | :--- | :--- |
| 1 | 帧格式/传输错误 | `frame_timeout`, `frame_too_long`, `invalid_frame` |
| 2 | 未知命令 | `unknown_command` |
| 3 | 参数缺失或非法 | `missing_parameter`, `parameter_out_of_range`, `unknown_field` |
| 4 | 内部执行失败 | `flash_write_error` |
| 6 | 未连接（未握手） | `not_connected` |

### 5.2. 产品扩展错误码
- 代码 5 及以上（除 6 外）保留给产品协议定义。产品协议应在其文档中声明使用的错误码及含义。
- 代码 1-4, 6 为通用错误码，所有产品一致。

---

## 6. 通用超时规范

| 命令 | 上位机超时 | 最大重试次数 | 说明 |
|:---|:---|:---|:---|
| handshake | 2000ms | 3 | 握手是后续通信的前提 |
| factory_reset | 5000ms | 1 | Flash 全擦可能耗时较长，不重试 |

产品特有命令的超时由产品协议定义。

**上位机超时处理**：达到最大重试次数后仍未收到有效应答，上位机应：
1. 提示用户连接异常
2. 尝试重新握手
3. 若握手也失败，断开连接并建议用户检查设备

## 7. 通用实现注意事项

### 7.1. MCU 实现要点
- 帧缓冲区大小由各产品根据其 `max_frame_len` 定义。基础协议默认 512 字节，产品可定义更大值（上限 8192 字节）。
- 实现帧同步状态机：IDLE → 等待 `[` → 接收中 → 收到 `]` 提交 / 超时丢弃。
- 帧接收超时定时器：1000ms。
- 解析时顺序扫描，按 `;` 分割为字段，对每个字段查找 `=` 分离键值。忽略空字段。
- 键值对存入固定大小数组（建议 20 个条目，大型产品适当扩容）。
- **严禁动态内存分配**，所有临时变量使用静态或栈内存。
- 维护连接状态机（IDLE / CONNECTED），在 IDLE 状态下拒绝配置管理命令（返回 code=6）。
- 命令处理器采用查表分发（命令名 → 处理函数），便于扩展新产品命令。

### 7.2. 上位机实现要点
- 握手后记录设备的 `max_frame_len`，后续发送的任何帧的数据部分不超过该值。
- 确保帧中无非法字符（`[`、`]`、`;`、`=`、空格）。
- 遵循串行命令模型：发一条 → 等应答 → 发下一条。
- 实现超时重试逻辑，按本协议及产品协议的超时规范执行。
- 收到错误码 6 (`not_connected`) 时自动重新握手。

### 7.3. 版本兼容
- 握手时交换 `base_proto_ver` 和 `biz_proto_ver`。上位机据此判断兼容性并调整通信策略。

### 7.4. 预留扩展
- **应用层 CRC**：未来版本可在帧末尾增加 `crc=<hex>` 字段，作为可选的完整性校验。
- **设备主动通知**：预留 `notify` 命令方向（设备→上位机），用于未来设备主动上报事件。
- **流模式**：基础协议在状态机中预留了流模式与连接超时的交互规则（见 3.2 节）。
