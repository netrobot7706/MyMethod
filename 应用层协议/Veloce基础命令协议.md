# Veloce 基础命令协议 v1.4

## 1. 概述

本文档定义 Veloce 系列 MIDI 控制器产品的**通用基础命令协议**。它规定了应用层帧格式、设备状态机、心跳保活、握手与恢复出厂等所有产品共用的机制。

基础协议基于 Jigsaw MIDI Transport 协议提供的透明字节流传输，采用定界帧内分号分隔的键值对明文编码。具体产品的业务命令由各产品的《产品命令协议》定义。

### 1.1. 设计目标
- **极简解析**：无需任何解析库，MCU 侧可用少量固定缓冲完成解码。
- **人类可读**：调试与日志记录直观，便于开发与现场排错。
- **紧凑高效**：避免 JSON 的结构冗余，适合低带宽 MIDI 传输。
- **可扩展**：新增参数不破坏老版本兼容性。
- **健壮**：具备帧同步恢复、超时处理、心跳保活，适应真实硬件环境。
- **复用**：基础协议可被所有后续产品继承，业务命令按产品定制。

### 1.2. 传输层依赖
- 传输层由《Jigsaw MIDI Transport 协议》提供可靠的双向字节流，最大帧长 8192 字节。应用层帧直接通过该字节流发送，无需额外封装。

### 1.3. 命令域划分
基础协议将命令分为两个域：

| 域 | 说明 | 版本不匹配时 |
|:---|:---|:---|
| **系统域** | 设备管理命令：`handshake`、`ping`、`factory_reset`、以及未来可能增加的 `enter_bootloader` 等 | **始终可执行**，不受协议版本匹配限制 |
| **配置域** | 产品业务命令（由产品协议定义） | 若 `base_proto_ver` 或 `biz_proto_ver` 主版本不匹配，**禁止执行** |

这样即使协议主版本升级导致配置域不兼容，用户仍可通过系统域命令完成固件升级（OTA）。

## 2. 帧格式

### 2.1. 定界与结构
- 每一帧以 **`[`** (ASCII 0x5B) 开始，以 **`]`** (ASCII 0x5D) 结束。
- 帧内包含一个或多个 **`key=value`** 对，各对之间用 **`;`** (ASCII 0x3B) 分隔。
- **首个字段必须为 `cmd=<命令名>`**。
- 字段顺序除 `cmd` 必须为首位外，其余不限。
- **最后一个字段后不加 `;`**。帧内不允许出现连续分号。
- 帧内**不允许**出现多余的空格、换行或任何不可见字符。
- **重复 key 禁止**：帧内同一 key 不得出现多次（如 `0_cc=70;0_cc=71`）。若出现，接收方应返回 `[cmd=error;code=3;msg=invalid_frame]`。

**合法示例**：
```
[cmd=handshake]
[cmd=handshake_ack;id=Veloce-A1B2;ver=1.0;base_proto_ver=1.4;biz_proto_ver=1.3;max_frame_len=512]
[cmd=ping]
```

### 2.2. 字符集与转义
- **允许字符**：大小写字母、数字、下划线 `_`、连字符 `-`、点 `.`、冒号 `:`、逗号 `,`、管道符 `|`。
- 键和值中**禁止**出现 `[`、`]`、`;`、`=` 这四个结构字符。
- **特别说明**：空格 (0x20) 不在允许字符集中。所有描述性文本必须使用下划线 `_` 替代空格。
- 设备 ID 等字符串由固件保证不包含上述禁止字符。上位机拼接数值时禁止在值中插入非法字符。
- **无转义机制**：协议不提供字符转义，以保持极简。

### 2.3. 帧长度限制
- **默认帧长**：基础协议定义默认最大帧数据长度（`[`...`]`之间）为 **512 字节**。
- **产品覆盖**：具体产品可在其《产品命令协议》中定义更大的帧长，上限不超过传输层最大帧长 **8192 字节**。
- **能力交换**：设备在握手时通过 `max_frame_len` 字段告知上位机本产品实际支持的最大帧长。上位机发送帧时不得超过该值。
- **溢出处理**：如果接收方在收到 `[` 后累计接收字节数超过 `max_frame_len` 仍未遇到 `]`，应将状态机置为 **IGNORE_UNTIL_BRACKET**——停止向缓冲区写入，静默丢弃后续字节，直到遇到 `]` 后再统一返回错误帧 `[cmd=error;code=1;msg=frame_too_long]`。此机制保护内存安全并防止后续帧错位。

### 2.4. 帧同步恢复机制
接收方在解析字节流时，必须遵循以下同步恢复规则：

1. **帧开始**：遇到 `[` 时，立即重置帧缓冲区，开始接收新帧。如果此前有未完成的帧（已收到 `[` 但未收到 `]`），则无条件丢弃未完成帧。如果当前处于 IGNORE_UNTIL_BRACKET 状态，遇到 `[` 时退出该状态并开始新帧。

2. **帧结束**：遇到 `]` 时，当前帧结束。如果处于 IGNORE_UNTIL_BRACKET 状态，则上报 `frame_too_long` 错误；否则将缓冲区内容提交给命令处理器，然后清空缓冲区。

3. **帧接收超时**：帧接收超时定时器（1000ms）在**所有接收相关状态下均生效**，包括正常接收状态和 IGNORE_UNTIL_BRACKET 状态。如果收到 `[` 后 1000ms 内未收到 `]`，则：
   - 退出 IGNORE_UNTIL_BRACKET 状态（若处于该状态）
   - 丢弃当前帧，清空缓冲区
   - 状态机回到 **IDLE（等待 `[`）**
   - 返回错误帧 `[cmd=error;code=1;msg=frame_timeout]`
   
   此机制确保即使数据流中断导致永远收不到 `]`，状态机也能自动恢复，不会永久卡死。

4. **上位机行为**：上位机若在发送帧后检测到连接中断或页面刷新，应在重连后重新握手，并丢弃任何未完成帧。

### 2.5. 数值约定
- 所有数值使用十进制 ASCII 字符串表示，不加前导零（除非值为 0，用 `0` 表示）。
- 当前协议所有数值均为非负整数。未来扩展如需支持负数，`-` 前缀表示负号。

## 3. 设备状态机

设备在应用层维护以下状态：

| 状态 | 说明 |
|:---|:---|
| **IDLE** | 上电初始状态。设备使用已保存配置（或出厂默认值）正常工作。等待上位机握手。 |
| **CONNECTED** | 上位机完成握手后的状态。设备可接受产品协议定义的配置域命令。 |
| **连接超时** | 在 CONNECTED 状态下，若 **30 秒内**未收到任何有效应用层报文（包括 `ping`），设备自动回到 IDLE 状态。若有流模式正在运行，先关闭流模式。下次上位机通信前需重新握手。 |

> **重要说明**：握手仅用于配置管理。设备在任何状态下都正常工作（发送 MIDI 数据等）。用户在不使用 Web App 的纯 MIDI 场景下无需握手。

### 3.1. IDLE 状态下收到配置域命令
在 IDLE 状态下，设备收到任何配置域命令时，应返回错误帧：

```
[cmd=error;code=6;msg=not_connected]
```

上位机收到错误码 6 后，应自动重新握手。

### 3.2. 心跳保活机制

为防止设备单方面进入 IDLE 导致上位机"静默断开"，引入心跳机制：

| 命令 | 方向 | 格式 | 说明 |
| :--- | :--- | :--- | :--- |
| ping | 上位机→设备 | `[cmd=ping]` | 心跳请求，保持连接活跃。 |
| pong | 设备→上位机 | `[cmd=pong]` | 心跳应答。 |

**上位机行为**：
- 在 CONNECTED 状态下，上位机应每隔 **10 秒**发送一次 `[cmd=ping]`。
- 每次发送 `ping` 后启动 **2000ms** 定时器等待 `pong`。如果在 2000ms 内收到 `pong`，重置连续失败计数器，并按 10 秒间隔发送下一个 `ping`。
- 若超时未收到 `pong`，连续失败计数 +1，并**立即重发**下一个 `ping`（无需等待 10 秒周期），直到连续失败达 3 次则判定连接断开，上位机应更新 UI 状态并提示用户。
- 等待 `pong` 期间，上位机必须忽略所有非 `pong` 及非 `error` 的异步帧（如 `report_adc`），不应中断等待或错误地将其视为应答。

**设备行为**：
- 收到 `ping` 立即回复 `pong`。收到 `ping` 视为收到有效应用层报文，刷新连接超时计时器。
- 若 30 秒内未收到任何上位机报文（包括 `ping`），判定连接超时，回 IDLE。

**流模式下的行为**：
- 流模式激活期间，连接超时检测**照常进行**。设备在流模式下仍应响应 `ping`。上位机在流模式下仍应每 10 秒发送 `ping`。
- 若上位机崩溃/关闭页面/断连，30 秒后设备超时，自动关闭流模式并回到 IDLE。这避免了流模式的死锁。

## 4. 通用命令

### 4.1. 命名规范
- 所有命令名采用 **`动词_名词`** 风格（如 `read_config`、`write_config`）。`handshake`、`ping`、`pong`、`factory_reset` 作为约定俗成术语保留。
- **`read` 类命令的应答帧**：对于请求帧**不携带业务参数**的 `read` 命令（如 `read_pot_config`），应答帧命令名与请求帧相同（`cmd=read_xxx`），通过帧内是否存在业务字段区分方向。对于请求帧**本身携带业务参数**的 `read` 命令（如 `read_adc`），应答可使用独立命令名（如 `report_xxx` 或 `read_xxx_rsp`）以明确区分方向。此例外在具体产品协议中明确。
- **`write`/`set`/`stream` 类命令的应答帧**：使用 `_ack` 后缀（如 `write_xxx_ack`），显式确认操作完成。命令执行成功由收到相应 `_ack` 帧本身表示，失败由 `error` 帧承载，`_ack` 帧内不再携带 `status` 字段。
- 所有命令和应答均遵循串行处理模型：**上位机应在收到当前命令的应答后再发送下一条命令**。设备不支持命令流水线。

> **串行模型与异步帧的交织说明**：本串行模型仅适用于上位机发起的请求-应答命令（如 `handshake`, `read_*`, `write_*`, `stream_adc`, `set_calibration` 等）。流模式数据（`report_adc`）和心跳（`ping`/`pong`）是异步行为，允许与命令应答帧交织。上位机在等待某命令应答时，必须按 `cmd` 字段过滤无关帧（如 `report_adc`、`pong`），继续等待目标应答，不被非目标帧打乱状态机。

---

### 4.2. 握手与设备识别
| 命令 | 方向 | 格式 | 说明 |
| :--- | :--- | :--- | :--- |
| handshake | 上位机→设备 | `[cmd=handshake]` | 发起应用层握手。 |
| handshake_ack | 设备→上位机 | `[cmd=handshake_ack;id=<设备ID>;ver=<固件版本>;base_proto_ver=<基础协议版本>;biz_proto_ver=<产品协议版本>;max_frame_len=<最大帧长>]` | 回应握手，提供身份、协议版本与能力信息。 |

**字段说明**：
- `id`：设备唯一标识，格式 `Veloce-<4字符大写十六进制>`，如 `Veloce-A1B2`。
- `ver`：固件主版本号（产品级别）。
- `base_proto_ver`：本基础协议版本号（当前 `1.4`）。基础协议版本变更是整个产品线的重大变更。
- `biz_proto_ver`：产品协议版本号（当前由具体产品文档定义）。上位机据此了解设备支持哪些产品特有命令。
- `max_frame_len`：本产品支持的最大帧数据长度（字节）。上位机发送任何帧时不得超过此值。取值范围：512 ~ 8192。

**版本兼容规则**：
- **基础协议**：若 `base_proto_ver` 主版本不匹配，上位机应提示用户更新固件/App。**系统域命令仍可正常使用**，配置域命令禁止执行。
- **产品协议**：若 `biz_proto_ver` 主版本不匹配，上位机应提示更新固件/App，配置域命令禁止执行。系统域命令不受影响。
- 次版本差异时，双方以较低次版本的功能集通信。

**超时**：上位机发送 `handshake` 后 **2000ms** 内未收到 `handshake_ack`，应视为超时。最多重试 3 次。

**示例**：
```
上位机 → 设备: [cmd=handshake]
设备 → 上位机: [cmd=handshake_ack;id=Veloce-A1B2;ver=1.0;base_proto_ver=1.4;biz_proto_ver=1.3;max_frame_len=512]
```

### 4.3. 恢复出厂设置
| 命令 | 方向 | 格式 | 说明 |
| :--- | :--- | :--- | :--- |
| factory_reset | 上位机→设备 | `[cmd=factory_reset]` | 清除所有用户配置，恢复出厂默认值并写入永久存储。 |
| factory_reset_ack | 设备→上位机 | `[cmd=factory_reset_ack]` | 恢复成功确认。 |

**执行时序**：设备必须在 Flash 操作**完全完成**后才发送 `factory_reset_ack`。上位机收到 ACK 后应延迟至少 **500ms** 再重新握手和读取配置。

**域归属**：系统域命令。不受协议版本匹配限制。

**超时**：上位机发送后 **5000ms** 内未收到应答，视为超时。不重试。

**示例**：
```
上位机 → 设备: [cmd=factory_reset]
设备 → 上位机: [cmd=factory_reset_ack]
```

## 5. 错误应答

当设备无法解析或执行命令时，返回错误帧：

```
[cmd=error;code=<错误码>;msg=<简短描述>]
```
`msg` 使用下划线替代空格，不得包含空格字符。

### 5.1. 通用错误码（基础协议）

| 代码 | 含义 | 示例 msg |
| :--- | :--- | :--- |
| 1 | 帧格式/传输错误 | `frame_timeout`, `frame_too_long`, `invalid_frame` |
| 2 | 未知命令 | `unknown_command` |
| 3 | 参数缺失或非法 | `missing_parameter`, `parameter_out_of_range`, `unknown_field`, `stream_active` |
| 4 | 内部执行失败 | `flash_write_error` |
| 6 | 未连接（未握手） | `not_connected` |

### 5.2. 产品扩展错误码
- 代码 5 及以上（除 6 外）保留给产品协议定义。产品协议应在其文档中声明使用的错误码及含义。
- 代码 1-4, 6 为通用错误码，所有产品一致。

---

## 6. 通用超时规范

| 命令 | 上位机超时 | 最大重试次数 | 说明 |
|:---|:---|:---|:---|
| handshake | 2000ms | 3 | 握手是后续通信的前提 |
| ping | 2000ms | 连续 3 次无 pong 则断连 | 见 3.2 节详细重试策略 |
| factory_reset | 5000ms | 1 | Flash 全擦可能耗时较长，不重试 |

产品特有命令的超时由产品协议定义。

**上位机超时处理**：达到最大重试次数后仍未收到有效应答，上位机应：
1. 提示用户连接异常
2. 尝试重新握手
3. 若握手也失败，断开连接并建议用户检查设备

## 7. 通用实现注意事项

### 7.1. MCU 实现要点
- 帧缓冲区大小由各产品根据其 `max_frame_len` 定义。基础协议默认 512 字节，产品可定义更大值（上限 8192 字节）。
- 实现帧同步状态机，包含三个状态：IDLE（等待 `[`）、RECEIVING（接收中）、IGNORE_UNTIL_BRACKET（溢出后忽略至 `]`）。帧接收超时定时器（1000ms）在 RECEIVING 和 IGNORE_UNTIL_BRACKET 状态下均生效，超时后无条件退出当前状态、清空缓冲区并回到 IDLE。
- **解析优化**：MCU 应首先提取并校验**第一个字段**的 `cmd` 值。若命令名不合法或不支持，立即停止后续字段解析并返回对应错误码（code=2），节省 MCU 算力。
- 合法字段的解析：顺序扫描，按 `;` 分割为字段，对每个字段查找 `=` 分离键值。忽略空字段。若检测到重复 key，立即返回 code=3。
- 键值对存入固定大小数组（建议 20 个条目，大型产品适当扩容）。
- **严禁动态内存分配**，所有临时变量使用静态或栈内存。
- 维护连接状态机（IDLE / CONNECTED），在 IDLE 状态下拒绝配置域命令（返回 code=6）。
- 命令处理器采用查表分发（命令名 → 处理函数），区分系统域和配置域。版本不匹配时禁止配置域命令。
- **防御性校验**：`read` 类命令的处理函数必须校验帧内是否包含业务数据字段（即应答格式才应有的字段）。若请求帧中包含了本应由设备上报的字段，MCU 应返回 error (code=3)，严禁当作正常请求处理。
- 实现 `ping`/`pong` 心跳机制，收到 `ping` 立即回复 `pong` 并刷新连接超时计时器。

### 7.2. 上位机实现要点
- 握手后记录设备的 `max_frame_len`，后续发送的任何帧的数据部分不超过该值。
- 确保帧中无非法字符（`[`、`]`、`;`、`=`、空格）。
- 遵循串行命令模型：发一条 → 等应答 → 发下一条。等待期间按 `cmd` 过滤异步帧（`report_adc`、`pong`），不被干扰。
- 实现超时重试逻辑，按本协议及产品协议的超时规范执行。
- 收到错误码 6 (`not_connected`) 时自动重新握手。
- **心跳机制**：每 10 秒发 `ping`，单次超时 2000ms 后立即重发，连续 3 次无 `pong` 判定断连。

### 7.3. 版本兼容
- 握手时交换 `base_proto_ver` 和 `biz_proto_ver`。上位机据此判断兼容性并调整通信策略。
- 主版本不匹配时，系统域命令仍可用，配置域命令禁止执行。这确保了 OTA 升级路径始终畅通。

### 7.4. 预留扩展
- **应用层 CRC**：未来版本可在帧末尾增加 `crc=<hex>` 字段，作为可选的完整性校验。
- **设备主动通知**：预留 `notify` 命令方向（设备→上位机），用于未来设备主动上报事件。
- **系统域命令扩展**：未来可增加 `enter_bootloader` 等系统域命令，始终不受版本匹配限制。
- **流模式频率协商**：产品协议可为 `stream_adc` 等流式命令增加 `interval_ms` 可选参数。
